= Design Patterns

The Mifos codebase draws on design patterns documented in the classic *_Design Patterns: Elements of Reusable Object-Oriented Software_* (Gamma, Helm, Johnson, Vlissides — the "`Gang of Four`") <<gof>> and the pragmatic wisdom of *_The Pragmatic Programmer_* (Hunt and Thomas) <<pp>>. Patterns are applied judiciously—only when they solve a real problem and make the code clearer, not as architectural decoration.

The patterns most commonly used in Mifos are:

**Builder Pattern** — For constructing complex objects with many optional parameters, such as loan products with dozens of configuration flags:

[source,java]
----
var loanProduct = LoanProduct.builder()
    .name("Short-Term Agricultural Loan")
    .shortName("AGRI-ST")
    .fundSource(fundSource)
    .principal(250000.00)
    .interestRate(12.5)
    .repaymentStrategy(RepaymentStrategy.PRINCIPAL_INTEREST)
    .amortization(Amortization.EQUAL_INSTALLMENTS)
    .interestCalculation(InterestCalculation.DECLINING_BALANCE)
    .allowPartialPeriodInterestCalculation(true)
    .daysInMonth(DaysInMonth.ACTUAL)
    .daysInYear(DaysInYear.ACTUAL)
    .build();
----

**Strategy Pattern** — For selecting business rule implementations at runtime, such as interest calculation methods:

[source,java]
----
public interface InterestCalculationStrategy {
    BigDecimal calculate(Loan loan, LocalDate asOfDate);
}

public class FlatInterestStrategy implements InterestCalculationStrategy {
    // ...
}

public class DecliningBalanceStrategy implements InterestCalculationStrategy {
    // ...
}

// Selected at runtime based on loan product configuration
var strategy = loanProduct.getInterestCalculation() == InterestCalculation.FLAT
    ? new FlatInterestStrategy()
    : new DecliningBalanceStrategy();
var interest = strategy.calculate(loan, asOfDate);
----

**Observer Pattern** — For domain events. When a loan is disbursed, multiple subscribers react independently: accounting creates entries, SMS notifications are sent, and audit logs are written.

**Repository Pattern** — For data access abstraction. All database interaction goes through repository interfaces, never directly through JPA entity managers or SQL strings in service code. This enables testing with in-memory implementations and swapping persistence technologies.
