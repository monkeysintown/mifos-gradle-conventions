= Secrets Management

== Password Policies

In a financial platform, the strength of authentication mechanisms directly impacts the security of customer data and funds. Mifos mandates the following password and authentication policies for all administrative and developer access:

**Multi-Factor Authentication (MFA)**: All accounts with access to production systems must use MFA. The preferred method is a **YubiKey** (FIDO2/WebAuthn hardware token), which provides phishing-resistant authentication. Hardware tokens are superior to SMS-based OTP because they cannot be intercepted by SIM-swap attacks.

**Passkeys and Passwordless Authentication**: Mifos is adopting passkeys (FIDO2/WebAuthn platform authenticators) for user-facing applications. Passkeys replace passwords with biometric or device-based authentication, eliminating the risk of credential theft through phishing or database breaches.

**Minimum password requirements** (when passwords are used):

* Minimum 12 characters
* Must include at least one uppercase letter, one lowercase letter, one digit, and one special character
* Must not match any of the last 12 passwords
* Must not appear in known breach databases (checked against HaveIBeenPwned)
* Maximum age: 90 days (for systems that still require passwords)

**YubiKey setup**:

[source,bash]
----
# Install YubiKey Manager
# macOS:
brew install ykman
# Linux:
sudo apt install yubikey-manager

# Check YubiKey status
ykman info

# Configure FIDO2 (WebAuthn) — this is used for GitHub and most services
ykman fido credentials list

# Generate a new FIDO2 credential for a service
ykman fido credentials add --rp-id github.com
----

For GitHub specifically, configure your YubiKey as a security key under *Settings → Password and authentication → Security keys*.

== Tools

=== Hashicorp Vault

Hashicorp Vault is the primary secrets management platform for Mifos production deployments. Vault provides secure storage, dynamic secret generation, encryption as a service, and fine-grained access control through policies.

Core Vault concepts used in Mifos:

* **KV Secrets Engine** (v2): Stores static secrets like database credentials, API keys, and TLS certificates. Versioned for audit trail.
* **PKI Secrets Engine**: Generates and manages TLS certificates for inter-service communication, eliminating manual certificate management.
* **Transit Secrets Engine**: Provides encryption-as-a-service for field-level data encryption without exposing encryption keys to applications.

Basic CLI operations:

[source,bash]
----
# Install Vault
brew install vault   # macOS
# or download from https://www.vaultproject.io/downloads

# Start a dev server (local development only)
vault server -dev

# Write a secret
vault kv put secret/mifos/database \
  username=fineract \
  password='s3cur3-p@ssw0rd!' \
  url=jdbc:postgresql://db:5432/fineract

# Read a secret
vault kv get secret/mifos/database
# ====== Data ======
# Key         Value
# url         jdbc:postgresql://db:5432/fineract
# username    fineract
# password    s3cur3-p@ssw0rd!

# Read a specific field
vault kv get -field=password secret/mifos/database

# List secrets
vault kv list secret/mifos/

# Generate a dynamic database credential (auto-expiring)
vault read mifos/database/creds/readonly
# Key             Value
# lease_id        mifos/database/creds/readonly/xxxx
# lease_duration  1h
# password        A1b2-C3d4-E5f6
# username        v-token-readonly-xxxx
----

Spring Boot integration pulls secrets directly from Vault at application startup:

[source,yaml]
----
# application.yml
spring:
  cloud:
    vault:
      uri: https://vault.mifos.org:8200
      authentication: KUBERNETES
      kv:
        enabled: true
        backend: secret
        application-name: mifos-fineract
----

=== OpenBao

OpenBao is the open-source fork of Hashicorp Vault, created in response to Vault's license change from MPL-2.0 to BSL-1.1. OpenBao maintains API compatibility with Vault, making it a drop-in replacement for organizations that require a fully open-source secrets management solution.

When to choose OpenBao over Vault:

* Your organization requires all infrastructure components to be under an OSI-approved open-source license.
* You are contributing to or deploying Mifos in contexts where BSL-1.1 is not acceptable.
* You want to participate in a community-governed secrets management project.

Installation:

[source,bash]
----
# Install OpenBao from source
git clone https://github.com/openbao/openbao.git
cd openbao
make bootstrap
make dev
./build/bao server -dev

# Or use the Docker image
docker run -d --name openbao \
  -p 8200:8200 \
  -e "BAO_DEV_ROOT_TOKEN_ID=root" \
  openbao/openbao:latest server -dev

# CLI is identical to Vault CLI (using 'bao' instead of 'vault')
bao kv put secret/mifos/database username=fineract password='s3cur3-p@ssw0rd!'
bao kv get secret/mifos/database
----

Because OpenBao is API-compatible with Vault, the Spring Boot Vault integration works with OpenBao by simply pointing the `spring.cloud.vault.uri` to the OpenBao server.

See:

- https://openbao.org/docs/install/

=== Bitwarden

Bitwarden is the recommended password manager for Mifos team members and is used for sharing non-production secrets (API keys for staging, test credentials, shared service accounts). It offers both a cloud-hosted and self-hosted option, with the self-hosted option preferred for sensitive deployments.

CLI usage for automation:

[source,bash]
----
# Install Bitwarden CLI
npm install -g @bitwarden/cli

# Log in (interactive)
bw login

# Unlock the vault (session key is needed for subsequent commands)
export BW_SESSION=$(bw unlock --raw)

# Search for a secret
bw get item "Mifos Staging DB"

# Get a specific field
bw get password "Mifos Staging DB"

# Create a new secure note
bw create item '{"type":2,"name":"New API Key","notes":"sk-xxxx","organizationId":"mifos-org"}'

# Sync the vault
bw sync

# List items in the Mifos organization collection
bw list items --organizationId mifos-org --collectionId staging
----

Bitwarden's organizational sharing features allow the Mifos team to create collections (e.g., "Production", "Staging", "Development") with granular access control. Team members can access only the secrets they need, and all access is logged.

=== 1Password

1Password is an alternative secrets manager preferred by some Mifos contributors for its polished developer experience and first-class CLI integration. The 1Password CLI integrates with environment variable injection, making it suitable for local development workflows.

Setup and usage:

[source,bash]
----
# Install 1Password CLI
# macOS:
brew install --cask 1password/tap/1password-cli
# Linux:
curl -sS https://downloads.1password.com/linux/keys/1password.asc | \
  sudo gpg --dearmor -o /usr/share/keyrings/1password-archive-keyring.gpg

# Sign in to your account
op account add --address mifos.1password.com --email dev@mifos.org
eval $(op signin)

# Retrieve a secret
op item get "Mifos API Key" --fields credential

# Inject secrets as environment variables
export DB_PASSWORD=$(op item get "Mifos Database" --fields password)

# Use with Docker (inject secrets into containers)
op run -- docker compose up

# CI/CD integration: use the 1Password GitHub Action
# .github/workflows/ci.yml:
# - uses: 1password/load-secrets-action@v2
#   with:
#     extract-variables: true
----

The 1Password service accounts feature enables CI/CD pipelines to access specific secrets without sharing the full vault, providing scoped access for automated workflows.

== Platforms

=== Github

The Mifos Initiative uses GitHub as its primary platform for source control, code review, and project management. The GitHub organization (https://github.com/mifos[mifos]) is structured to provide clear ownership and governance across the project portfolio.

**Organization structure**:

* **Core team**: Maintainers with admin access to all repositories.
* **Module owners**: Contributors with maintain access to specific repositories.
* **Contributors**: Anyone with write access after onboarding.
* **Community**: All other GitHub users who can open issues and submit PRs.

**Repository naming conventions**:

* `fineract` — The core banking engine
* `community-app` — The reference web application
* `mobile-wallet` — The mobile SDK
* `fineract-cn-[module]` — Fineract CN (cloud-native) modules
* `mifos-x-[module]` — Mifos X platform extensions

**Branch protection rules** on `main`:

* Require signed commits
* Require PR reviews (at least 1 approving review)
* Require status checks to pass (build, test, security scan)
* Require linear history (no merge commits)
* Dismiss stale reviews on push
* Restrict who can push to matching branches

**Required status checks**:

* `build` — Gradle build succeeds
* `test` — All tests pass
* `checkstyle` — Code style is compliant
* `spotless` — Code formatting is correct
* `security-scan` — No critical/high vulnerabilities
