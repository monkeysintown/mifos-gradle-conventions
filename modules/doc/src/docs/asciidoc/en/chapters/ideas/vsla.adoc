= Self-Funded Groups (VSLA) Service

Domain-Driven Design & Event-Based Architecture plan.

== 1. Executive Summary & Architecture Topology

The VSLA (Self-Funded Group) module is implemented as a **standalone Spring Boot application** (`vsla-service`) that sits alongside a standard Apache Fineract deployment. Fineract Core is treated as an external financial ledger and account engine; it is **not modified** and is consumed exclusively through the **official Apache Fineract Java REST API Client** (`fineract-client`).

=== Key Principles

* **VSLA Service is the System of Record** for all VSLA-specific domain logic: group parameters, meeting schedules, attendance, internal lending lifecycle, voting, share-out calculations, earnings allocation, and group-level accounting.
* **Fineract Core is the System of Record** for: client registry, savings account balances, external loans (group ↔ bank/MFI), and institution-level GL postings.
* **All cross-system mutations** are asynchronous and eventually consistent, using an **Outbox + Inbox pattern** over the Fineract REST API.
* **No shared database**; loose coupling via REST only.
* **Trapdoor setting** (`vsla.group-type`) is a deployment-time configuration in the VSLA service. When active, all group lifecycle and internal lending operations are owned by `vsla-service`; Fineract remains unaware of group semantics.

== 2. Integration Strategy with Fineract Core

=== 2.1 Official Fineract Java REST API Client

The VSLA service consumes the `fineract-client` JAR (autogenerated from Fineract's OpenAPI spec, Retrofit2/Feign/Spring Web Client/-based).

[source,java]
----

**Service Account Principle:** A dedicated Fineract user (`vsla-service`) is created with minimal permissions: `CREATE_CLIENT`, `CREATE_SAVINGSACCOUNT`, `SAVINGS_ACCOUNT_TRANSACTIONS`, `CREATE_LOAN`, `LOAN_TRANSACTIONS`, `CREATE_JOURNALENTRY`, `READ_REPORT`.

| **Users / Offices** | `GET /users`, `GET /offices` | Read-only sync for user-group mapping validation and office hierarchy.
|===

=== 2.3 Outbox Pattern for Eventually Consistent Integration

IMPORTANT: Let's see if we can pull it off initially by just using Spring application events. For resilience the outbox pattern is the preferred solution, but instead of a fixed scheduler a CDC solution like Debezium could be used for real-time processing.

Because the VSLA service database and Fineract are not enrolled in a distributed transaction, all outbound REST calls to Fineract are deferred through an **Outbox** table within the VSLA database. This guarantees at-least-once delivery with idempotency.

[source,java]
----
@Entity
@Table(name = "outbox_events")
public class OutboxEvent {
    @Id private UUID id;
    private String aggregateType;   // e.g., "GroupCapital"
    private String aggregateId;   // group UUID
    private String eventType;     // e.g., "SharesPurchased"
    private String destination;   // e.g., "fineract.savings.deposit"
    @Column(columnDefinition = "jsonb") private String payload;
    @Enumerated(STRING) private OutboxStatus status; // PENDING, PROCESSED, FAILED
    private int retryCount;
    private Instant createdAt;
    private String errorMessage;
}
----

**Flow:**

1. Command handler updates VSLA aggregate and appends a domain event (same local TX).
2. An event listener inserts an `OutboxEvent` row (same local TX).
3. A background `@Scheduled` processor polls `PENDING` rows, executes the Fineract REST call via `fineract-client`, and marks `PROCESSED` or `FAILED`.
4. Fineract APIs use `externalId` (or idempotency keys derived from `OutboxEvent.id`) to prevent duplicate financial postings on retries.

[source,java]
----
@Component
@RequiredArgsConstructor
public class FineractOutboxProcessor {

    private final OutboxRepository outboxRepo;
    private final FineractClient fineract;
    private final Map<String, FineractCommandHandler> handlers;

    @Scheduled(fixedDelayString = "${vsla.outbox.poll-ms:5000}")
    @Transactional
    public void processPending() {
        List<OutboxEvent> pending = outboxRepo.findTop100ByStatusOrderByCreatedAtAsc(PENDING);
        for (OutboxEvent event : pending) {
            try {
                handlers.get(event.getDestination()).handle(event);
                event.markProcessed();
            } catch (Exception ex) {
                event.fail(ex.getMessage()); // dead-letter after max retries
            }
        }
    }
}
----

== 3. Strategic Design — Bounded Contexts (Inside VSLA Service)

The VSLA service itself is organized as vertical slices (bounded contexts) communicating via domain events.

[cols="1,2,2",options="header"]
|===
| Bounded Context | Responsibility | Integration with Fineract

| **Group Lifecycle** | Group creation, cycle dates, meeting schedules, member enrollment, operational parameters (shares, penalties, loan products) | Creates/updates Fineract `Client` records for members via Outbox.
| **Capital & Shares** | Mandatory share purchases, member savings tracking, corpus computation, share-out estimation | Orchestrates Fineract savings account creation and deposit/withdrawal transactions via Outbox.
| **Internal Lending** | Member loans from pooled funds, eligibility rules, voting approval, repayment scheduling, interest/fine calculation | **Does NOT create Fineract Loan accounts**. Tracks loans in VSLA DB; uses Fineract `accounttransfers` or `savingsaccounts/transactions` to move actual funds.
| **External Funding** | Group borrowing from banks/MFIs, government grants, revolving funds | Creates Fineract `Loan` accounts (group as borrower) and tracks repayment schedules via Outbox.
| **Group Accounting** | Group-level COA, trial balance, P&L, balance sheet, earnings allocation, bonus deposits, investments | Maintains own GL in VSLA DB. Optionally posts summary journal entries to Fineract Accounting API for institutional consolidation.
| **Membership & Governance** | Attendance, lateness, voting power, grading/loyalty, user-to-group mapping | Read-only sync with Fineract Users API; enforces VSLA-specific authorization rules.
| **Mobile & Payments** | Offline sync, YAPE/PLIN webhooks, mobile dashboards, conflict resolution | Receives webhooks directly; translates to VSLA commands and Outbox entries for Fineract savings deposits.
|===

=== Context Map

* **Group Lifecycle** → upstream context publishing `SelfFundedGroupCreated`, `MemberEnrolled`, `MeetingHeld`.
* **Capital & Shares** → downstream consumer of Group Lifecycle; customer-supplier to Fineract Savings API.
* **Internal Lending** → downstream consumer of Capital events; uses Fineract Account Transfers API for fund movements.
* **Group Accounting** → published-language subscriber to all monetary domain events; produces no Fineract side-effects except optional summary JE postings.

== 5. New Command Processing (Inside VSLA Service)

IMPORTANT: Re-use fineract-command library and maybe extend it to support "queries" (aka the “Q” in CQRS).

The VSLA service (re-)uses the **type-safe command dispatcher** (aka "new command processing"). Commands are immutable; handlers use business logic services to that load aggregates, invoke domain behavior, persist, and publish events.

NOTE: We could try to drop the service concept altogether and use as single concern pure business logic services.

=== 5.3 Example Handler

[source,java]
----
@Service
@RequiredArgsConstructor
public class SharesServiceImpl implements SharesService {
    private final SelfFundedGroupRepository groupRepo;
    private final GroupCapitalRepository capitalRepo;

    // ...

    @Override
    public SharesPurchaseResponse purchase(SharesPurchaseRequest request) {
        var group = groupRepo.findById(command.groupId())
            .orElseThrow(() -> new GroupNotFoundException(command.groupId()));

        var capital = capitalRepo.findByGroupId(command.groupId());

        // domain behavior
        group.validateMeetingIsOpen(command.meetingId());
        capital.creditShares(command.memberId(), command.numberOfShares(), command.amount());

        // persist
        groupRepo.save(group);
        capitalRepo.save(capital);

        return SharesPurchaseResponse.builder().groupId(command.groupId().value()).build();
    }

    // ...
}

@Component
@RequiredArgsConstructor
public class SharesPurchaseCommandHandler implements CommandHandler<SharesPurchaseRequest, SharesPurchaseResponse> {

    private final SharesService sharesService;
    private final EventPublisher eventPublisher;

    @Override
    public SharesPurchaseResponse handle(Command<SharesPurchaseRequest> command) {
        var response = sharesService.purchase(command.getPayload());

        // publish internal domain event
        eventPublisher.publish(new SharesPurchased(
            command.groupId(), command.memberId(), command.numberOfShares(),
            command.amount(), command.meetingId()
        ));

        return response;
    }
}

=== 6.2 Event Listeners → Outbox

[source,java]
----
@Component
@RequiredArgsConstructor
public class FineractIntegrationEventListener {

    private final OutboxRepository outboxRepo;
    private final FineractMappingService mappingService;

    @EventListener
    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    public void onSharesPurchased(SharesPurchased event) {
        FineractAccountMapping mapping = mappingService.findSavingsMapping(event.groupId(), event.memberId());

        var payload = new SavingsAccountTransactionPostRequest()
            .transactionDate(LocalDate.now().toString())
            .transactionAmount(event.totalAmount().getAmount())
            .paymentTypeId(1) // configured payment type for share purchase
            .locale("en")
            .dateFormat("yyyy-MM-dd");

        var outbox = OutboxEvent.builder()
            .id(UUID.randomUUID())
            .aggregateType("GroupCapital")
            .aggregateId(event.groupId().value())
            .eventType("SharesPurchased")
            .destination("fineract.savings.deposit")
            .payload(JsonUtils.write(payload))
            .status(PENDING)
            .build();

        outboxRepo.save(outbox);
    }

    @EventListener
    @TransactionalEventListener(phase = TransactionPhase.BEFORE_COMMIT)
    public void onExternalFundingReceived(ExternalFundingReceived event) {
        // Maps to Fineract Loan application for group as borrower
        // LoansPostRequest payload = ... ;
        outboxRepo.save(OutboxEvent.builder()
            .destination("fineract.loan.create")
            .payload(payload)
            .build());
    }
}

== 7. Data Ownership & Transaction Boundaries

=== 7.1 Data Ownership Matrix

[cols="2,2,2",options="header"]
|===
| Data | Owner | Storage

| Group parameters, meetings, attendance, votes | VSLA Service | VSLA DB
| Internal loan schedules, repayments, eligibility rules | VSLA Service | VSLA DB
| Share ledger, corpus calculations, share-out estimates | VSLA Service | VSLA DB
| Group-level Chart of Accounts, GL entries, TB/P&L/BS | VSLA Service | VSLA DB
| Member master data (name, KYC, office) | Fineract Core | Fineract DB
| Savings account balances & transactions | Fineract Core | Fineract DB
| External loan balances & schedules | Fineract Core | Fineract DB
| Institution-level GL / journal entries | Fineract Core | Fineract DB
|===

=== 7.2 Transaction Model

* **VSLA Local Transaction:** A command handler updates one aggregate (and its entities) within a single Spring `@Transactional` boundary. The Outbox row is written in the same transaction.
* **No Distributed Transactions:** Fineract REST calls happen asynchronously via the Outbox processor. If a Fineract call fails, the Outbox row is retried with exponential backoff until a dead-letter threshold is reached.
* **Compensating Actions:** For irreversible Fineract operations (e.g., loan disbursement), the Outbox processor first calls Fineract, then on success updates a `fineract_sync_log` in VSLA. If the VSLA transaction fails after Fineract succeeded, a reconciliation job detects the orphan and creates a compensating Outbox entry (e.g., reverse the journal entry).

=== 8.2 Idempotency against Fineract

Every Outbox payload that mutates Fineract state must include an `externalId` derived from the Outbox event UUID:

[source,java]
----
var req = new SavingsAccountTransactionPostRequest()
    .externalId(outboxEventId.toString()) // Fineract stores external_id uniquely per transaction type
    // .transactionDate(...)
    // .transactionAmount(...)
    ;
----

This ensures that retries of the same Outbox event are idempotent in Fineract.

=== 8.3 Reconciliation Job

A nightly Spring Batch / `@Scheduled` job compares:

* VSLA `GroupCapital` computed corpus vs. sum of linked Fineract savings account balances.
* VSLA internal loan outstanding principal vs. net savings transfers in Fineract.
* Discrepancies produce `ReconciliationAlert` domain events for ops review.

== 9. Offline Mobile & Payment Integration

=== 9.1 Mobile-First REST API

The mobile app consumes **VSLA Service APIs only**. It never talks directly to Fineract.

* `GET /vsla/groups/{id}/sync?since={sequence}` — pull server events since last sync.
* `POST /vsla/commands/batch` — push offline-queued commands.
* `GET /vsla/groups/{id}/dashboard` — CQRS read model.

=== 9.2 Offline Command Flow

1. Mobile app records commands locally (SQLite) with `clientCommandId` and `deviceTimestamp`.
2. On connectivity, mobile **pulls** new VSLA domain events first, then **pushes** its batch.
3. `OfflineBatchCommandHandler`:
* Validates each command against current aggregate version.
* Detects conflicts (e.g., savings withdrawal recorded offline while web transaction already processed).
* Applies merge strategy: **cash-in** accepted tentatively; **cash-out** rejected if conflict.
* Publishes `OfflineDataMerged` event with resolution log.

=== 9.3 Payment Webhooks (YAPE / PLIN)

Payment providers POST directly to the VSLA service:

[source,java]
----
@RestController
@RequestMapping("/vsla/webhooks")
@RequiredArgsConstructor
public class PaymentWebhookController {

    private final CommandDispatcher dispatcher;

    @PostMapping("/yape")
    public ResponseEntity<Void> receiveYape(@RequestBody YapeWebhookPayload payload) {
        var command = new MobilePaymentReceiveCommand(
            GroupId.of(payload.getGroupReference()),
            PaymentGateway.YAPE,
            payload.getTransactionId(),
            Money.of(payload.getAmount(), Currency.getInstance("PEN")),
            Instant.now()
        );
        dispatcher.dispatch(command);
        return ResponseEntity.accepted().build();
    }
}
----

The command handler:

1. Validates idempotency via `externalTransactionId`.
2. Credits `GroupCapital` (domain event).
3. Outbox listener creates Fineract savings deposit for the group pooled account.
4. Mobile push listener notifies the group's active device.

== 13. Critical Design Decisions

[cols="1,2",options="header"]
|===
| Decision | Rationale

| VSLA as standalone Spring Boot app | Fineract Core remains unmodified, upgrade-safe, and the VSLA team can iterate on its own release cycle. The official `fineract-client` provides a stable, type-safe integration boundary.
| Internal loans NOT mapped to Fineract Loan accounts | Fineract's loan module assumes the institution is the lender. VSLA internal loans (group → member) are tracked in the VSLA domain; only fund movements use Fineract Savings / Account Transfer APIs.
| Outbox pattern for all Fineract mutations | Eliminates distributed transactions and ensures eventual consistency. Fineract's `externalId` fields make retries idempotent.
| Group-level accounting in VSLA DB | Fineract GL is institution-scoped. Group-level COA, TB, P&L, and BS are native to the VSLA domain and stored locally for fast dashboard queries.
| Typed command bus inside VSLA service | Provides a clean DDD application layer, middleware extensibility, and aligns with Fineract's own command-processing evolution without coupling to its internal `JsonCommand` infrastructure.
| Cash-in accepted offline, cash-out rejected offline | Enforced as domain invariants in `GroupCapital`: cash-out requires confirmation of latest server-side Fineract savings balance (optimistic locking via sequence numbers).
| Mobile app talks only to VSLA service | Decouples the mobile frontend from Fineract's complex API and allows VSLA-specific offline sync semantics without proxying every Fineract endpoint.
|===

== 14. Summary

This plan delivers a **type-safe, event-driven, DDD-compliant** VSLA service that:

* Stands **independently** alongside Apache Fineract,
* Uses the **official Fineract Java REST API client** (`fineract-client`) as its sole integration mechanism,
* Implements its own **typed command bus** and **domain event model**,
* Guarantees eventual consistency with Fineract via the **Outbox pattern**,
* Maintains **group-level accounting** natively while delegating client registry, savings, and external loans to Fineract Core,
* Supports **offline-first mobile** workflows and **YAPE/PLIN** payment webhooks as first-class integration events.

The result is a loosely coupled, horizontally scalable, and upgrade-safe architecture that respects Fineract as the financial backbone while giving the VSLA domain the autonomy it requires.
