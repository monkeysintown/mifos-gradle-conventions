== Methodology & Metrics

Based on industry-standard software architecture metrics (Robert C. Martin's package metrics, 5C-Design principles, and dependency-depth measures), the following metrics are applied:

|===
| Metric | Symbol | Formula | Meaning

| Efferent Coupling | Ce | Count of feature packages this package depends on | Fan-out; outgoing cross-feature dependencies
| Afferent Coupling | Ca | Count of feature packages that depend on this package | Fan-in; incoming cross-feature dependencies
| Instability | I | Ce / (Ca + Ce) | 0 = maximally stable (depended-upon), 1 = maximally unstable (leaf)
| Cross-Feature Violations | CFV | Count of dependencies ≠ fineract-core/command | Violations of the target modular boundary rule
| Dependency Depth | DD | Longest path in cross-feature dependency graph | How deep transitive coupling reaches
| Transitive Coupling | TC | Total direct + indirect cross-feature dependencies | Full blast radius of a change
| Independence Score | IS | 1 − (CFV / (N−1)) where N = total features | 1.0 = fully independent, 0.0 = depends on everything
|===

=== Key Structural Decisions in this Gantt:

1. *Parallel Foundation (Phase 1):* *1a* (fineract-core) and *1b* (fineract-command) are set to start simultaneously since they are foundational module creations that don't depend on each other.
2. *Sequential Low-Hanging Fruit (Phase 2):* Step *2a* (Extracting DTOs) must happen first. Once the DTOs are in fineract-core, the decoupling of the near-independent packages (*2b* and *2c*) can proceed.
3. *Critical Path (Phase 3):* *3a* (Savings) must be decoupled first because both *3b* (Loan) and *3c* (Deposits) heavily depend on it. Once Savings is decoupled, Loan and Deposit refactoring can run in parallel since they are independent of each other at that point.
4. *Parallel Orchestrators (Phase 4):* Teller, Report, Share, and Batch decoupling can all happen in parallel once the core domain packages (Loan, Savings, Deposits) are stable. I've split them to start at the end of their respective Phase 3 dependencies.
5. *Final Gate (Phase 5):* ArchUnit rules and CI/CD enforcement are strictly sequenced at the very end to lock down the architecture once all refactoring is complete.

== Key Takeaways

. **6 of 20 packages** (charge, calendar, rate, organisation, notifications, template) are already fully compliant — no refactoring needed
. **4 packages** (loan, savings, fixeddeposit, recurringdeposit) account for **73% of all violations** and are the critical refactoring targets
. The **deposit→savings inheritance** is the deepest coupling (DD=4) and requires the most architectural change (composition over inheritance + lifecycle interface in core)
. The **teller** and **report** packages are cross-cutting orchestrators that should be refactored to command-bus and CQRS patterns respectively
. Moving **read-only DTOs** and **domain event/command definitions** into `fineract-core` resolves ~60% of violations with relatively low effort
. All cross-feature communication should flow through `fineract-command` (event bus / command bus) — never via direct service injection


== Non-Trivial Cross-Feature Communication: Loan Disbursement

One of the most complex, non-trivial cross-feature interactions in Fineract is **Loan Disbursement**.

When a loan is disbursed, it currently triggers a cascade of direct service calls across multiple domains:

. **Loan domain:** Updates the loan status to "Active" and generates the repayment schedule.
. **Accounting domain:** Creates double-entry journal entries (Debit Loan Portfolio, Credit Bank/Fund Account).
. **Savings domain:** (If linked) Deposits the disbursed amount into the client's savings account.
. **Charge domain:** Deducts upfront charges/fees from the disbursed amount.
. **Notification domain:** Sends a notification to the client.

Currently, `LoanWritePlatformService` is bloated with `@Autowired` dependencies from all these other feature packages. If any downstream step fails, the monolithic `@Transactional` rolls back everything, creating tight coupling and massive transaction scopes.

Here is how we refactor this to flow exclusively through `fineract-command` using **Domain Events** and the **Transactional Outbox Pattern** (to handle the non-trivial distributed data consistency challenge).

=== The Core Problem: The Current Anti-Pattern

[source,java]
----
// CURRENT: fineract-loan directly importing other features
@Service
public class LoanWritePlatformService {
    @Autowired private LoanAccountRepository loanRepository;
    @Autowired private JournalEntryWritePlatformService accountingService; // Cross-feature!
    @Autowired private SavingsAccountWritePlatformService savingsService;  // Cross-feature!
    @Autowired private ChargeWritePlatformService chargeService;          // Cross-feature!

    @Transactional
    public void disburseLoan(Long loanId, DisbursementCommand command) {
        Loan loan = loanRepository.findById(loanId);
        loan.disburse(command); // 1. Loan domain logic

        accountingService.createJournalEntries(loan); // 2. Direct call
        chargeService.deductUpfrontCharges(loan);     // 3. Direct call

        if (loan.isLinkedToSavings()) {
            savingsService.deposit(loan.getSavingsId(), loan.getAmount()); // 4. Direct call
        }
    }
}
----

=== The Solution: Event-Driven Choreography via `fineract-command`

We shift from **synchronous orchestration** to **asynchronous choreography**. The Loan domain only cares about disbursing the loan. It announces this fact to the world. The other domains react independently.

==== Step 2.1: Define the Event in `fineract-command`

Events are plain POJOs/Records defined in `fineract-command` so all features can reference them without depending on each other's implementations.

[source,java]
----
// TARGET: fineract-command (accessible by all features)
package org.apache.fineract.command.events;

import java.math.BigDecimal;
import java.time.LocalDate;

public record LoanDisbursedEvent(
    Long loanId,
    Long officeId,
    Long savingsId, // Nullable
    String currencyCode,
    BigDecimal principal,
    BigDecimal upfrontCharges,
    LocalDate disbursementDate
) {}
----

==== Step 2.2: Refactor `fineract-loan` to Publish the Event

We remove the cross-feature dependencies. To ensure the database update and the event publishing happen atomically (non-trivial part!), we use the **Transactional Outbox Pattern**. Instead of publishing directly to a message broker (which can fail and cause data inconsistency), we save the event to an Outbox table in the *same* database transaction.

[source,java]
----
// TARGET: fineract-loan (no cross-feature imports)
@Service
public class LoanWritePlatformService {
    @Autowired private LoanAccountRepository loanRepository;
    @Autowired private EventOutboxRepository outboxRepository; // In fineract-core
    // REMOVED: accountingService, savingsService, chargeService

    @Transactional
    public void disburseLoan(Long loanId, DisbursementCommand command) {
        Loan loan = loanRepository.findById(loanId);
        loan.disburse(command); // 1. Pure Loan domain logic
        loanRepository.save(loan);

        // 2. Map to event
        LoanDisbursedEvent event = new LoanDisbursedEvent(
            loan.getId(),
            loan.getOfficeId(),
            loan.getLinkedSavingsId(),
            loan.getCurrency().getCode(),
            loan.getPrincipal(),
            loan.sumUpfrontCharges(),
            loan.getDisbursementDate()
        );

        // 3. Save to Outbox (guarantees atomicity with loan update)
        outboxRepository.save(EventOutboxEntry.from(event));
    }
}
----
*A background relay (provided by `fineract-command` infrastructure) polls the outbox table and publishes the events to the Command/Event Bus (e.g., Kafka, or an in-process Spring ApplicationEventPublisher for a monolith).*

==== Step 2.3: `fineract-accounting` Subscribes to the Event

The Accounting module is now completely independent. It contains an Event Listener that processes the event and creates the journal entries.

[source,java]
----
// TARGET: fineract-accounting (no cross-feature imports)
@Component
public class AccountingLoanEventHandler {

    @Autowired private JournalEntryWritePlatformService journalEntryService;
    @Autowired private EventOutboxRepository outboxRepository;

    @Transactional
    @EventHandler // Custom annotation recognized by fineract-command relay
    public void handle(LoanDisbursedEvent event) {
        // Idempotency check! (Crucial for async events)
        if (journalEntryService.isJournalEntryExists("LOAN_DISBURSEMENT", event.loanId())) {
            return;
        }

        // Pure Accounting domain logic
        JournalEntry entry = JournalEntry.builder()
            .debit(Accounts.LOAN_PORTFOLIO, event.principal())
            .credit(Accounts.FUND_SOURCE, event.principal().subtract(event.upfrontCharges()))
            .credit(Accounts.INCOME_FROM_CHARGES, event.upfrontCharges())
            .officeId(event.officeId())
            .currency(event.currencyCode())
            .build();

        journalEntryService.createEntry(entry);

        // Mark event as processed in outbox/inbox to guarantee exactly-once processing
        outboxRepository.markProcessed(event);
    }
}

=== Addressing the Non-Trivial Challenges

When adopting this architecture, two non-trivial problems emerge that `fineract-command` must solve:

The Dual-Write Problem:: You cannot safely update a database and publish to a message broker in the same step. If the broker is down, the data is inconsistent.
+
*Solution:* The **Transactional Outbox** pattern shown above. `fineract-loan` writes its state and the event to the *same database*. `fineract-command` provides the background relay that reads the outbox and pushes to the bus.

Eventual Consistency & Idempotency:: In the current monolith, if accounting fails, the whole transaction rolls back. In the new model, if Accounting processes the event, but Savings is temporarily down, the system is in a temporarily inconsistent state. Furthermore, the relay might deliver the event twice ("at-least-once delivery").
+
*Solution:* Subscribers (like `AccountingLoanEventHandler`) **must be idempotent**. Before creating the journal entry, they check if a transaction for `loanId = X` and `type = DISBURSEMENT` already exists. If it does, they safely ignore the duplicate event.

Sagas for Compensating Transactions:: What if the Loan disburses successfully, but the Accounting module realizes the GL accounts are frozen and cannot accept the entry?
+
*Solution:* `fineract-command` must support **Saga patterns**. Accounting would publish an `AccountingEntryFailedEvent`. The Loan module would subscribe to this event and trigger a compensating transaction (`ReverseDisbursementCommand`) to revert the loan status.
