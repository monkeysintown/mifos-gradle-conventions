[glossary]
= Development

ADR:: Architecture Decision Record — a document capturing an important architectural decision along with its context and consequences. ADRs are stored in the repository under `docs/adr/` and follow the Michael Nygard format: a short title, a status (proposed, accepted, deprecated, superseded), context describing the forces at play, the decision itself, and the consequences of that decision. ADRs are immutable once accepted; if a decision is reversed, a new ADR is created that supersedes the original. In the Mifos project, ADRs are required for any change that affects the public API, the data model, the deployment architecture, or the technology stack.

BDD:: Behavior-Driven Development — a development approach using Gherkin-style specifications (Given/When/Then) to define expected behavior before implementation. BDD bridges the gap between business stakeholders and developers by expressing requirements in a structured, executable format. In the Mifos platform, Cucumber is used to implement BDD acceptance tests for critical business processes such as loan origination, account opening, and payment processing. These tests serve as both specification and verification, ensuring that the implemented behavior matches the business intent.

CQRS:: Command Query Responsibility Segregation — a pattern that separates read and write operations for a data store. In a traditional CRUD architecture, the same model handles both reads and writes, which can lead to models that serve neither purpose well. CQRS addresses this by defining separate models for commands (writes) and queries (reads), allowing each to be optimized independently. The Mifos Fineract platform applies CQRS in the command processing pipeline, where commands are validated, authorized, and processed through the write model, and queries are served by read-optimized projections. This separation enables the write model to enforce business rules and invariants while the read model provides efficient, denormalized views for reporting and user interfaces.

DDD:: Domain-Driven Design — a software design approach focusing on modeling software to match a domain, based on input from domain experts. DDD provides both strategic patterns (Bounded Context, Context Map, Ubiquitous Language) for managing the complexity of large domains and tactical patterns (Entity, Value Object, Aggregate, Domain Event, Repository) for implementing individual bounded contexts. The Mifos platform's modular architecture is organized around bounded contexts (Loans, Savings, Accounting, Portfolio), each with its own domain model and ubiquitous language that reflects the terminology used by financial inclusion practitioners.

IaC:: Infrastructure as Code — managing and provisioning infrastructure through machine-readable definition files rather than manual processes. IaC applies software engineering practices—version control, code review, automated testing, continuous integration—to infrastructure management, ensuring that environments are reproducible, auditable, and self-documenting. The Mifos project uses Terraform (or OpenTofu) for infrastructure provisioning and Ansible for configuration management, with all definitions stored in version-controlled repositories. See <<infrastructure>> for the complete infrastructure-as-code strategy.

SemVer:: Semantic Versioning — a versioning scheme using MAJOR.MINOR.PATCH format to communicate the nature of changes. Given a version number MAJOR.MINOR.PATCH: increment MAJOR when you make incompatible API changes, increment MINOR when you add functionality in a backward-compatible manner, and increment PATCH when you make backward-compatible bug fixes. Pre-release versions (e.g., `1.2.0-alpha.3`) and build metadata (e.g., `1.2.0+build.456`) extend the basic format. The Mifos release pipeline enforces SemVer automatically through semantic-release, which derives version numbers from Conventional Commits. See <<releases>> for details.

TDD:: Test-Driven Development — a development methodology where tests are written before the implementation code. The TDD cycle follows three steps: write a failing test (Red), write the minimum code to make the test pass (Green), and then improve the code while keeping the test passing (Refactor). This cycle is often abbreviated as Red-Green-Refactor. TDD ensures that every line of production code has a corresponding test and that the codebase remains testable as it evolves. In the Mifos project, TDD is recommended for core domain logic where correctness is paramount (loan calculations, interest accrual, account balancing).
