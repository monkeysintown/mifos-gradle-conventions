[glossary]
= General

CI/CD:: Continuous Integration / Continuous Delivery — practices of frequently integrating code changes and automating the delivery pipeline. Continuous Integration requires developers to merge their changes into the shared repository frequently (at least daily), with each merge triggering an automated build and test suite to detect integration errors early. Continuous Delivery extends CI by ensuring that every successful build produces a release-ready artifact that can be deployed to production with a single approval. The Mifos CI/CD pipeline is implemented in GitHub Actions and follows the pipeline-as-code model, where the workflow definitions are version-controlled alongside the application code.

PR:: Pull Request — a mechanism for proposing code changes and requesting review before merging into the main development branch. In the Mifos workflow, PRs are the primary unit of code review and serve as a permanent record of the discussion and decision-making that accompanied each change. Every PR must pass automated CI checks, receive at least one approved review, and conform to the Conventional Commits format before it can be merged. PRs targeting the `main` branch require two approvals and signed commits.

SBOM:: Software Bill of Materials — a formal record of the components and dependencies used to build a software artifact. An SBOM lists every library, framework, and tool included in the build, along with their versions, licenses, and provenance. SBOMs are increasingly required by regulations (such as the U.S. Executive Order on Improving the Nation's Cybersecurity) and are essential for vulnerability management: when a security advisory is published for a specific library version, an SBOM allows you to quickly determine whether your deployment is affected. The Mifos Gradle build generates an SBOM using CycloneDX as part of the release process.
